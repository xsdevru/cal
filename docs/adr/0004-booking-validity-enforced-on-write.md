# Валидность брони гарантируется на записи

Инвариант: бронь всегда стоит на реальном свободном слоте владельца. Держим его на сервере,
а не доверяем фронту.

- **Доступность — на владельца.** Занятость считается по *всем* активным броням владельца
  (статусы `accepted`/`pending`) по всем типам встреч, а не по одному `eventTypeId`. Один
  физический владелец не может вести две встречи одновременно.
- **Проверка на записи через слоты.** `Bookings_create`/`reschedule` принимают время, только
  если оно *точно совпадает* со свободным слотом, который вернёт расчёт слотов на тот день
  (переиспользуем `server/src/lib/slots.ts` как единственный источник истины — не дублируем
  правила доступности). При переносе бронь исключается из расчёта занятости (не конфликтует
  сама с собой).
- **Отказ — `409 Conflict`** с телом `Error` (единый код на все причины «не свободный слот»:
  занято/вне доступности/прошлое/мимо сетки; различие — в `message`).

## Consequences

Между проверкой и вставкой есть окно гонки (два гостя на один слот). Для MVP на одного
владельца (см. [ADR-0002](./0002-single-owner-mvp-no-auth.md)) принимаем его осознанно:
трафик мал, SQLite сериализует записи. Робастная защита (интерактивная транзакция с
`BEGIN IMMEDIATE` либо ограничение БД) отложена до появления реальной конкуренции.

# Политика подтверждения на уровне типа встречи

Подтверждение брони — свойство типа встречи, а не глобальная настройка. У `EventType`
появляется поле `confirmation` (расширяемое перечисление, не булев флаг):

- `auto` — новая бронь сразу `accepted` (текущее поведение, значение по умолчанию);
- `manual` — новая бронь создаётся `pending` и ждёт решения владельца;
- `payment` — **зарезервировано**, не реализовано (подтверждение оплатой).

Владелец разрешает `pending` двумя операциями (симметрично `cancel`): `POST
/bookings/{uid}/accept` → `accepted`, `POST /bookings/{uid}/reject` → `rejected`. Валидны
только из `pending`, иначе `409`.

Машина состояний: create → `accepted`(auto)|`pending`(manual); `pending` → `accepted`|`rejected`
(владелец); `accepted`/`pending` → `cancelled` (гость) или reschedule (статус не меняется);
`cancelled`/`rejected` терминальны.

## Consequences

`pending`-бронь **держит слот эксклюзивно** (входит в занятость, см.
[ADR-0004](./0004-booking-validity-enforced-on-write.md)): пока владелец не решит, слот занят
для остальных. Ненадёжный гость может держать слот — снимается тем, что владелец reject'ит.
Авто-истечения `pending` (TTL) в MVP нет: нет фонового планировщика, а «истёкший» был бы
пятым статусом. TTL логично ввести вместе с `payment`. Перечисление выбрано вместо булева
флага, чтобы `payment` добавлялся одной вариантой без комбинаторики флагов.

Требует правок контракта: поле `confirmation` в `models.tsp`, операции accept/reject в
`bookings.tsp`, и UI владельца для подтверждения/отклонения в разделе броней.
